# フェーズ完了確認 — `$ARGUMENTS`

このコマンドは [docs/ai_context/prompt_templates.md](../../docs/ai_context/prompt_templates.md) の TPL-007 を実行可能なチェックリスト形式に変換した skill です（phaseX-11 Step 4）。
予防対象：[AP-007 設計書の機能を部分実装で完了報告](../../docs/ai_context/ai_collaboration_antipatterns.md#ap-007-設計書の機能を部分実装で完了報告)

> **対象フェーズ：`$ARGUMENTS`**
> 以下のチェックリストを上から順に実施し、すべて埋まってから完了報告してください。
> 引数が空の場合は、対象フェーズをユーザーに確認してから進めてください。

---

## チェック 1：設計書の機能一覧 vs 実装状況の○×表

`docs/design/phaseX/$ARGUMENTS*.md` または `docs/design/phase*/$ARGUMENTS*.md` を読み、
記載されている **画面・API・DB・運用機構** をすべて抽出し、実装状況を○×で報告してください。

| 種別 | 項目 | 設計書の記述 | 実装ファイル | 状況 |
|------|------|------------|------------|------|
| 画面 | ... | ... | ... | ○/× |
| API  | ... | ... | ... | ○/× |
| DB   | ... | ... | ... | ○/× |
| 運用 | ... | ... | ... | ○/× |

× が 1 つでもあれば、その時点で完了報告は不可。実装漏れを潰してから次へ。

## チェック 2：プレースホルダー残置の grep

`$ARGUMENTS` の対象範囲で、以下のパターンが残っていないかを grep で検査し、結果を報告してください：

- `フェーズ.*で実装` / `フェーズ.*で対応` / `フェーズXで実装予定` 等
- `TODO` / `FIXME` / `未実装` / `WIP`
- Vue / React コンポーネントの空テンプレート（`<template>\s*<div>\s*</div>` 等の本文不在）

検出された残置は、設計書のスコープ外（明示的に次フェーズへ申し送り済み）か否かで分類し、
スコープ内なら完了報告前に解消してください。

## チェック 3：実機ブラウザ確認

対象画面をブラウザで開き、ログイン → 主要フロー 1 周を実行した結果を記述してください。
AI が直接ブラウザ操作できない場合は **「ユーザー側で確認が必要」と明示** し、
ユーザーに確認手順を提示してください。

確認内容（最低限）：

- [ ] ローカル環境（`docker compose up`）で対象画面を開いて主要フローが動く
- [ ] 認証絡み・Cookie 絡み・画像配信絡みの機能は **本番ドメインで** 通す（030/031/032/033 起因）
- [ ] スクリーンショットを取得し、PR 説明文に添付する手順を提示

## チェック 4：CI グリーン単独では完了としない

`CI が緑 = フェーズ完了` ではないことを明示してください。
チェック 1〜3 のすべてが揃って初めて「完了」とする。

CI で確認する項目（参考）：

- [ ] PHPUnit / JUnit テストがすべてグリーン
- [ ] `mvn test -Dsurefire.runOrder=random` で分離が壊れていないことが確認済み（amazia-core 側、AP-009 関連）
- [ ] デプロイ機構の健全性（`deploy.yml` リカバリ + カナリアが過去 N 回 Failed なし、022/024/025/026 起因）

## チェック 5：DB / API 設計書の同フェーズ更新

`$ARGUMENTS` で DB スキーマ変更や API エンドポイント追加を行った場合、
[docs/ai_context/design_doc_maintenance.md](../../docs/ai_context/design_doc_maintenance.md) のルールに沿って
以下が **当該フェーズ内で** 更新されているかを確認してください：

- [ ] `docs/database_design/TBL_*.md`（新規テーブル分の定義書）
- [ ] `docs/database_design/README.md`（ファイル一覧表）
- [ ] `docs/database_design/ER_diagram.md`（Mermaid 図のリレーション）
- [ ] `docs/api_design/Core_API.md` / `Console_API.md` / `Market_API.md`
- [ ] `ops/healthcheck/required_tables.txt`（Core 主要テーブル追加時）

更新漏れがあれば、それは「フェーズ完了」ではなく次フェーズへの負債。
このフェーズ内で潰してください（P12〜P14 で 14 テーブル / 29 エンドポイントが未文書化のまま放置された前例あり）。

## チェック 6：test_insights.md「まとめ」との整合

[docs/ai_context/test_insights.md](../../docs/ai_context/test_insights.md) の §まとめ「フェーズ完了の定義」
（10 項目）を上から順に照合し、満たしているもの・満たせないものをそれぞれ列挙してください。

特に注意すべき項目：

- 4. 環境変数：`docker-compose.yml` / `phpunit.xml` / `.env.example` の **3 点セット** 更新
- 6. Docker 初回起動：`docker compose down -v && docker compose up --build` でエラーなく起動する（018/029 起因）
- 9. 本番環境での実機確認（ローカルだけでは Cookie/JWT/プロキシの不整合は検知できない）

## 最終報告フォーマット

以下のテンプレートで、ユーザーに完了判定の最終報告を提示してください：

```markdown
## $ARGUMENTS 完了確認レポート

### 1. 機能一覧 vs 実装状況
<○×表をここに>

### 2. プレースホルダー残置
- 検出: <件数>件 / 解消済: <件数>件

### 3. 実機ブラウザ確認
- ローカル: <結果 or ユーザー確認待ち>
- 本番: <結果 or ユーザー確認待ち>

### 4. CI 状況
- 単体テスト: <結果>
- 分離テスト（random order）: <結果>
- デプロイ機構: <直近 N 回の状況>

### 5. DB/API 設計書更新
- DB 変更ありなし: <あり/なし>
- 更新済ファイル: <列挙>
- 未更新（負債）: <列挙 or なし>

### 6. test_insights.md「まとめ」との整合
- 満たした項目: 1, 2, 3, ...
- 未充足項目: <なし or 列挙>

### 判定
- [ ] 完了として OK
- [ ] 残課題あり：<列挙>
```

> - skill が呼び出されたこと自体（`$ARGUMENTS` の値含む）を最終報告の冒頭に記載し、TPL-007 の置き換えで実施したことを明示してください
> - チェックリストの項目で対象外（スコープ外明示など）がある場合は、設計書のどの節で除外されているかを引用してください
