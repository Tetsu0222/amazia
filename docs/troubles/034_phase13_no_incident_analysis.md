# 034: フェーズ13 トラブル0件の不在分析（メタ記録）

## ステータス
🟡 様子見（2026-05-06 時点で観測上は無事故 / 観測網の死角を継続監視）

## 発症箇所
- 該当なし。Amazia Market 認証フロー（フェーズ13）の設計・実装・ローカル / 検証環境動作確認まで、単体テスト以外で表面化した不具合がない

## 症状
- 観測された症状はない
- ただし、これだけ広範囲（DB マイグレーション 5 テーブル / セッション認証 Filter 自作 / CSRF Filter 自作 / 顧客 API 5 種 / SES 連携 / 郵便局 CSV 取込 / Market フロント 5 画面 / CloudFront Behavior 追加）の実装で `docs/troubles/` への新規記載が 0 件というのは、過去フェーズ（11・12 でも複数件発生）と比較して構造的に異常
- 「故障が起きていない」のか「故障に気づく口がない」のかが切り分けできていない状態そのものを記録として残す

## 根本原因（仮説の整理）

### 仮説A：過去の蓄積が事前回避を効かせた（ポジティブ）
- `docs/ai_context/test_insights.md` の 12 カテゴリと 033 までのトラブル知見が、設計段階で地雷を踏まないコースを敷いた
- 具体的に効いたと思われる対応：
  - 027（H2 / MySQL DDL 互換）→ phase13 §11.3 で migration を H2 に流さない方針を採用
  - 019・020（Cookie domain・Set-Cookie 中継）→ §3.2 で `MARKET_COOKIE_SECURE` / `MARKET_COOKIE_DOMAIN` を環境変数化、Console 経由の中継ではなく CloudFront 直 Behavior で /api/customer/* を分離
  - 018（Docker 初回起動 + JPA Entity 不在）→ §11.5 で `down -v` 起動確認をフェーズ完了条件に明記
  - 009（環境変数の管理漏れ）→ §11.4 に新規環境変数のセット更新ルールを明記

### 仮説B：本番運用の薄さで故障が露出していない（ネガティブ）
- 実ユーザーは開発者 1 名、SES は数通、`market_sessions.expires_at` の GC・`failed_attempts` ロック解除・パスワード再発行トークンの `used` フラグなどは検証回数が少ない
- 並行ログイン・長時間放置・大量登録などの状況で出る故障は構造的に観測不能
- 「無料枠完走」方針上、負荷試験は意図的にスコープ外

### 仮説C：観測網の死角（最も警戒すべき）
- 故障がカウントされていないだけで、サイレント故障は起きうる
- 観測されていない / 観測手段が薄い領域：
  - `market_sessions` の TTL 切れレコードの GC（手動 DELETE が必要かどうか確認していない）
  - `failed_attempts` がロック解除後に 0 リセットされているか（CloudWatch メトリクスなし）
  - SES バウンス・コンプレイント通知（SNS Topic 未設定の可能性）
  - CSRF 検証失敗（403）の件数集計
  - パスワード再発行トークンの未使用期限切れ件数
- これらは「ログを見に行けば分かる」程度の解像度で、能動的に集計するダッシュボードはまだない

### 比率の見立て
- A : B : C ≒ 4 : 3 : 3
- A は確実に効いているが、B・C の寄与を切り分けない限り「設計が成熟した」のか「観測解像度が低いだけ」かは判別不能

## なぜ CI で検知できなかったか
- そもそも検知すべき故障が特定できていない（不在の分析という性質上）
- CI（JUnit / Vitest）は単体・結合の正常系と既知の異常系を見ているだけで、運用観測は対象外
- 観測網の死角（仮説C）は CI の責務ではなく、本番モニタリング側の課題

## 修正内容
- 修正そのものは不要
- 代わりに本ドキュメントを「不在の分析」として残し、後続フェーズで類似の故障が出たときに、phase13 時点で何を見ていなかったかを起点に切り分けられるようにする
