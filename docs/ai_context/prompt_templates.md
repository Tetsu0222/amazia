# プロンプトテンプレート集

## 目的

作業種別ごとに、AI へ渡す定型プロンプトを集約する。
本ファイルのテンプレを使うことで、[ai_collaboration_antipatterns.md](ai_collaboration_antipatterns.md) の
AP-* に該当する典型的な落とし穴を着手前に予防できる。

## 使い方

1. これから着手する作業の種別に対応する TPL を本ファイルから選ぶ
2. テンプレ中の `<...>` 部分を実際の値に置き換える
3. AI への最初の指示として渡す（または既存指示文に併記する）

テンプレは「AI に何を確認・報告させてから本実装に入るか」を定めるもの。
着手前のチェックを省略せず、確認結果を踏まえて実装方針を提示させる構造になっている。

### TPL-001: 新規テーブル追加時

**予防する AP：** [AP-001 既存スキーマ・既存DDLを読まずに新規追記](ai_collaboration_antipatterns.md#ap-001-既存スキーマ既存ddlを読まずに新規追記)

**いつ使うか：** 新規テーブルを Core の `schema.sql` に追加する前。FK 列を含むテーブルなら必須。

**テンプレ：**

> `<テーブル名>` を追加する前に、以下を確認・報告してください：
>
> 1. `amazia-core/src/main/resources/schema.sql` を読み、参照先となる既存テーブル
>    （特に users.id・他の参照先となる ID 列）の型を抽出
> 2. 新規テーブルの FK 列が同一型（BIGINT UNSIGNED など）になっているか
> 3. 既存に同概念のテーブルがないか grep（`grep -i <概念名> schema.sql`）
> 4. `docs/database_design/` の既存 `TBL_*.md` を確認し、命名規則・FK 命名・index 規約と整合するか
>
> 確認結果を踏まえ、実装方針（DDL 案・更新が必要な周辺ファイル一覧）を提示してから
> 実装に入ってください。
>
> 関連 AP: AP-001

**関連する周辺ファイル更新（参考）：**
- `amazia-core/src/main/resources/schema.sql` — DDL 追記
- `docs/database_design/TBL_<テーブル名>.md` — 新規定義書
- `docs/database_design/README.md` — ファイル一覧表
- `docs/database_design/ER_diagram.md` — Mermaid 図
- `ops/healthcheck/required_tables.txt` — Core 主要テーブルなら追加

### TPL-002: ライブラリ・API 採用時

**予防する AP：** [AP-004 API/SDK のシグネチャ・仕様を確認せず外挿](ai_collaboration_antipatterns.md#ap-004-apisdk-のシグネチャ仕様を確認せず外挿)

**いつ使うか：** 新規ライブラリの採用、外部 API の利用、内部 API（Core ↔ Console / Console ↔ Market）の参照を含む実装前。

**テンプレ：**

> `<ライブラリ名 / API名>` の `<機能>` を使う前に、推測ではなく実コード／公式ドキュメントの
> 引用で以下を報告してください：
>
> **API（内部 DTO 参照）の場合：**
> - 該当 DTO クラスのファイルパス
> - 全フィールド名（getter 名）と型
> - リスト API と詳細 API のフィールド差分（軽量版／フル版で構造が違うことが多い）
>
> **ライブラリの場合：**
> - 採用するメソッド／クラスの正確なシグネチャ
> - デフォルト挙動（特にデフォルトで無効化されている設定）
> - 制約（鍵長・型・バージョン依存・スレッドセーフ性など）
>
> 上記を踏まえ、実装方針を提示してから実装に入ってください。
>
> 関連 AP: AP-004

### TPL-003: テストヘルパー作成・改修時

**予防する AP：** [AP-002 Entity / 制約を読まずにテストヘルパーをハードコード](ai_collaboration_antipatterns.md#ap-002-entity--制約を読まずにテストヘルパーをハードコード)

**いつ使うか：** Java/Spring Boot 側のテストヘルパー（Entity を生成するファクトリメソッド）を作成・改修する前。

**テンプレ：**

> `<Entity名>` 用のテストヘルパーを作成・改修する前に、Entity クラスを読み、以下を報告してください：
>
> 1. クラス／フィールドに付与されている `@UniqueConstraint` `@Column(unique=true)` `@Index(unique=true)` の有無
> 2. 制約カラムが存在する場合、ヘルパーで複数インスタンスを作る際の一意化戦略
>    （引数化／`System.nanoTime()` 付与／カウンタ など）
> 3. 既存に同 Entity 用のヘルパーがあれば、その実装パターンとの整合
>
> 上記を踏まえ、ヘルパーのシグネチャと一意化方針を提示してから実装に入ってください。
>
> 関連 AP: AP-002

### TPL-004: マイグレーション機構の確認

**予防する AP：** [AP-003 存在しない技術スタックを仮定（Flyway 誤認）](ai_collaboration_antipatterns.md#ap-003-存在しない技術スタックを仮定flyway-誤認)

**いつ使うか：** 新フェーズで DB スキーマ変更を伴う作業に着手する前。または既存スキーマの本番反映方法を確認したいとき。

**テンプレ：**

> 新フェーズの実装計画を立てる前に、以下を確認・報告してください：
>
> 1. `amazia-core/pom.xml` の dependency にマイグレーション機構（flyway-core / liquibase 等）が含まれるか
> 2. 含まれない場合、本番でスキーマが流れる経路は何か
>    - `amazia-core/src/main/resources/application*.properties` の `spring.sql.init.mode` の値
>    - `amazia-core/src/main/resources/schema.sql` の存在・サイズ
> 3. `db/migration/V*.sql` のファイル群が本番で実行されるか否か（実行されない場合は名残ファイル）
> 4. Console 側（Laravel）のマイグレーションパス（`amazia-console/database/migrations/`）の運用状況
>
> 確認結果を踏まえてから、新規スキーマ変更の方針（schema.sql に追記するのか、Console 側マイグレーションか）を提案してください。
>
> 関連 AP: AP-003

### TPL-005: シェル / SQL 文字列構築時

**予防する AP：** [AP-005 シェル / SQL のクォート・エスケープを読まず実装](ai_collaboration_antipatterns.md#ap-005-シェル--sql-のクォートエスケープを読まず実装)

**いつ使うか：** `.github/workflows/*.yml` の SSM コマンド、`docker exec ... sh -c '...'` を含むコマンド、シェル経由で SQL を流すスクリプトを実装・改修する前。

**テンプレ：**

> `sh -c '<INNER>'` または `docker exec ... sh -c` を含むコマンドを実装する前に、以下を整理して報告してください：
>
> 1. ホスト bash → コンテナ sh の各層で何が展開されるか（変数・クォート）
> 2. `<INNER>` 中に含まれる以下の特殊文字が、どの層で問題になり得るか
>    - `'` （シングルクォート）— 外側との衝突
>    - `"` （ダブルクォート）— 同上
>    - `$` — 変数展開タイミング
>    - `\` — エスケープ
> 3. 採用するクォート戦略
>    - ホスト変数を保護したい場合：外側シングルクォート
>    - コンテナ内変数を展開したい場合：外側ダブルクォート
> 4. SQL リテラルの `'` を含む場合の `'\''` エスケープの要否
>
> 上記を整理してから最終コマンドを提示し、合わせて「ホストでこのコマンドが echo されたとき
> どう見えるか」「コンテナで実行されるときどう見えるか」を併記してください。
>
> 関連 AP: AP-005

### TPL-006: バグ修正時の横展開確認

**予防する AP：** [AP-006 対症療法で本質根因を残す](ai_collaboration_antipatterns.md#ap-006-対症療法で本質根因を残す)

**いつ使うか：** 不具合の原因が「ある概念の判定基準・データ参照先・エラーハンドリング」にある場合。同概念が複数箇所に分散している可能性がある修正で必須。

**テンプレ：**

> `<不具合の症状>` を修正する前に、以下を実施・報告してください：
>
> 1. 同概念のロジック（例：判定基準・データ参照先・エラーハンドリング）が他のファイルに存在しないか grep で確認
>    - 検索キーワード候補：`<キーワード1>`, `<キーワード2>`
> 2. 見つかった全箇所の現状ロジックを列挙し、修正対象と「直さない箇所」の判断根拠を提示
> 3. 修正完了の判断基準を「症状解消」ではなく「根因の単一化・横展開確認完了」とする
>
> 修正後の報告では「直した箇所」と「直さなかった箇所の根拠」を併記してください。
> 「動いた」だけで完了とせず、根因の単一化が達成されているかを必ず述べてください。
>
> 関連 AP: AP-006

### TPL-007: フェーズ完了確認時

**予防する AP：** [AP-007 設計書の機能を部分実装で完了報告](ai_collaboration_antipatterns.md#ap-007-設計書の機能を部分実装で完了報告)

**いつ使うか：** フェーズの完了報告（design/phase*.md のステータスを「完了」にする）をする前。

**テンプレ：**

> `<フェーズ番号>` の完了報告をする前に、以下を実施してください：
>
> 1. 設計書（`docs/design/phase*/<該当ファイル>.md`）に記載された画面・API・DB を一覧化し、それぞれの実装状況を○×で報告
> 2. プレースホルダー（`"フェーズ.*で実装"` `"TODO"` `"未実装"` 等の残置）が無いか grep
> 3. 対象画面をブラウザで開き、全フローを 1 周実行した結果を記述（または「ユーザー側で確認が必要」と明示）
> 4. CI グリーンだけでは完了とせず、上記 3 つすべてが揃って初めて「完了」とする
> 5. DB / API 設計書（`docs/database_design/` `docs/api_design/`）の更新が当該フェーズ内で済んでいるか確認
>
> 上記を満たさない状態で完了報告しないでください。フェーズ完了の定義は
> `docs/ai_context/test_insights.md` の「まとめ」セクションも合わせて参照してください。
>
> 関連 AP: AP-007

### TPL-008: CI/CD・運用機構導入時

**予防する AP：** [AP-008 CI/CD・運用設計の検証不足で副次トラブル誘発](ai_collaboration_antipatterns.md#ap-008-cicd運用設計の検証不足で副次トラブル誘発)

**いつ使うか：** GitHub Actions ワークフロー、SSM 関連スクリプト、ヘルスチェック、リカバリ機構、デプロイ手順などを実装・改修する前。

**テンプレ：**

> CI/CD・運用機構（リカバリ・ヘルスチェック・自動化スクリプト等）を実装・改修する前に、以下を報告してください：
>
> 1. この機構が「何を検証していて」「何を保証していないか」を明文化
>    - 例：PingStatus=Online は「ハートビート」のみで「コマンド配信可否」は別
> 2. 関連する過去トラブルを `docs/troubles/` から grep し、知見を引用
>    - SSM 系：003/008/022/023/024/025/026/028
>    - スキーマ系：027/037/038/044/045/049
>    - シェル/SQL 系：046/048
> 3. 本番でしか再現しない挙動を検証するためのカナリア戦略の有無
> 4. 失敗時の検知経路（ログ・通知・リトライ上限・手動介入のトリガー）
>
> 「動いた」だけで完了とせず、上記の保証範囲・検知経路を明示してから実装してください。
> 実装後は故意失敗テスト（schema から 1 テーブル削除する等）で検知が機能するかを必ず検証してください。
>
> 関連 AP: AP-008

## 運用ルール

### 新規 TPL の追加基準

- 既存 TPL-001〜008 でカバーできない作業種別が現れたとき
- 対応する AP が `ai_collaboration_antipatterns.md` に存在することが前提
- AP と TPL は 1 対 1 ではなく N 対 N でよい（1 つの作業種別が複数 AP を予防することがある）

### AP との同期

`ai_collaboration_antipatterns.md` の AP-* を更新したら、対応する TPL も併せて更新する。
特に AP の「対応プロンプトスニペット」と TPL の本文は内容を同期させる
（TPL は作業種別の汎用形、AP のスニペットは特定 AP の最小形という位置付け）。

### テンプレの陳腐化チェック

半年に 1 回程度、`docs/troubles/` の最新トラブルを棚卸しし、TPL のテンプレ文が
現状の実装と整合しているか確認する。例えば：

- マイグレーション機構が変わったら TPL-004 を更新
- DTO 構造が大きく変わる方針になったら TPL-002 を更新
- フェーズ完了の定義が変わったら TPL-007 を更新
