# 不具合対応時のトラブルドキュメント・テンプレート

## このファイルの位置付け

ユーザーから不具合・障害の調査・修正を依頼された場合に、AI が **必ず** 実施すべき
ドキュメント作業の手順とテンプレートを定める。CLAUDE.md からポインタ参照される本体ファイル。

phaseX-11 Step 2 で CLAUDE.md 本文（直書き 30 行超）から切り出し。

## いつ参照するか（トリガー条件）

- ユーザーから「不具合」「障害」「バグ」「動かない」「想定と違う」等の文言で調査・修正を依頼されたとき
- 既存 `docs/troubles/NNN_*.md` の更新を依頼されたとき
- AP-006（対症療法で本質根因を残す）・AP-001/002/004/005（着手前確認系）に該当する作業に着手する直前

## 不具合対応時に必ず行う作業

ユーザーから不具合・障害の調査・修正を依頼された場合、コード修正に加えて以下を**必ず**行う。

### トラブルドキュメントの作成・更新

1. `docs/troubles/` にある既存ファイルを確認し、同一・類似の問題がすでに記録されていないか確認する
2. 新規の不具合であれば `docs/troubles/NNN_<概要>.md` を新規作成する（NNN は連番）
3. 既存ファイルに追記すべき内容があれば更新する
4. 修正完了後、`docs/ai_context/test_insights.md` に新たなテスト観点が抽出できる場合は追記する
5. 修正完了後、`docs/ai_context/ai_collaboration_antipatterns.md` の既存 AP-* に該当する場合は出典欄に追加。該当しない新規パターンが現れた場合は AP-NNN として追記を検討する
6. 新規作成時、不具合対応完了時、`docs/troubles/README.md` の更新も忘れない。

### 同根トラブルの横展開確認（重要）

AP-001/005/006/008/009 には「直前のトラブル修正で類似クラスの再発を見落とす」共通根がある
（[ai_collaboration_antipatterns.md](ai_collaboration_antipatterns.md) §横断観点）。

不具合修正時は「同じパターンが他のクラス・ファイル・エンドポイントに無いか」を必ず grep し、
横展開で潰せるものはまとめて修正する。単発修正で閉じない。

### 派生節の活用（連番乱立抑制）

同根の不具合は新規連番を切らず、親トラブルに「派生節」として追記する運用とする
（feedback_trouble_doc_consolidation の方針）。新規 NNN 連番化は「根因が異なる」と判断できた場合のみ。

## ドキュメントに記載する内容（テンプレート）

```
# NNN: <タイトル>

## ステータス
🔴 調査中 / 🟡 対応中 / ✅ 解決済（YYYY-MM-DD）

## 発症箇所
<URL・画面名・エンドポイントなど>

## 症状
<ユーザーが観察した現象>

## 根本原因
<調査で判明した原因>

## なぜ CI で検知できなかったか
<テストで防げなかった理由>

## 修正内容
<実施した変更の概要>

## 再発防止
| 観点 | 対策 |
|------|------|
| ... | ... |

## AI協働観点
- AI の判断ミス：<あれば記載／無ければ「該当なし」>
- 人間が止めるべきだった点：<あれば記載／無ければ「該当なし」>
- 該当アンチパターン：AP-NNN <or 新規パターンとして追加 / なし>
```

`AI協働観点` 節は「AI 協働で何が落ちたか」を継続蓄積するためのもの。判定の3軸と詳細は
[ai_collaboration_antipatterns.md](ai_collaboration_antipatterns.md) の前提節を参照。
本質的にはすべて人間側の責任である前提のもと、改善のレバーを見つけるための便宜的な
切り分けとして記録する。

## AP-NNN 関連表（不具合対応時の AP 参照ガイド）

不具合対応時に「該当アンチパターン」欄を埋めるための逆引き表。各 AP の詳細・対応プロンプト
スニペットは [ai_collaboration_antipatterns.md](ai_collaboration_antipatterns.md) と
[prompt_templates.md](prompt_templates.md) を参照。

| 不具合の典型パターン | 該当 AP | 対応 TPL |
|----------------------|---------|----------|
| 既存スキーマ・既存 DDL を読まずに新規追記してしまった | AP-001 | TPL-001 |
| Entity / 制約を読まずにテストヘルパーをハードコードした | AP-002 | TPL-003 |
| 存在しない技術スタックを仮定（例：Flyway 誤認） | AP-003 | TPL-002 |
| API/SDK のシグネチャ・仕様を確認せず外挿した | AP-004 | TPL-004 |
| シェル / SQL のクォート・エスケープを読まず実装した | AP-005 | TPL-005 |
| 対症療法で本質根因を残した | AP-006 | TPL-006 |
| 設計書の機能を部分実装で完了報告した | AP-007 | TPL-007 |
| CI/CD・運用設計の検証不足で副次トラブルを誘発した | AP-008 | TPL-008 |
| テスト分離不足 + 単発 PR で類似クラスを見落とした | AP-009 | TPL-009 |

**双方向リンクの運用：**
- AP の出典欄に troubles/NNN を追記し、troubles 側の「該当アンチパターン」欄に AP-NNN を書く
- 同根の他箇所が見つかった場合は派生節として親トラブルに追記し、AP の出典欄もその旨を更新
