## 📅 日次バッチ

### ① 商品入荷数と在庫数の整合性チェック
- 入荷テーブルと在庫テーブルを照合  
- 不一致がある場合、対象商品の担当権限者へ通知  
- 通知方法：メール or Console の通知センター（要検討）

---

### ② 発売日・予約ステータスの自動更新
- 発売日・公開日・予約開始日の条件に基づき、商品ステータスを更新  
- フェーズ14・フェーズ15の仕様に基づく  
- 対象ステータス例  
  - 予約開始前  
  - 予約受付中  
  - 公開中  
  - 在庫切れ（予約可）  
- 各フェーズの仕様を再確認し、ロジックを統合する必要あり

---

### ③ 売上と在庫数の照合
- 売上テーブルと在庫テーブルを比較  
- 不一致がある場合、権限者へ通知  
- 売上金額の集計も実施  
- 振込確認の疑似API  
  - ランダム関数で「一致 / 不一致」を返す  
  - 一致 → 正常  
  - 不一致 → NG として通知

---

### ④ 配送ステータスの自動更新
- 配送予定日・発送日・配達完了日を基にステータスを更新  
- トラブル関数（後述）により、一定確率で配送遅延・不在再配達を発生させる

# ⚠ トラブル関数（シミュレーション）

運用テストのため、ランダム関数を用いて「トラブル発生」を再現する。

### ① 売上不一致関数
- 一定確率で売上金額と振込金額を不一致にする  
- 不一致時は権限者へ通知  
- テスト目的で確率は調整可能（例：5%）

### ② 在庫数不一致関数
- 一定確率で在庫数と売上数を不一致にする  
- 不一致時は通知  
- 在庫管理の異常系テストに利用

### ③ 配送トラブル関数
- 宛先不在  
- 配送遅延  
- 再配達  
- 配送ステータスを自動で変更し、Market の購入履歴にも反映

# 実行基盤の比較検討

ポートフォリオの「思考プロセス」を成果物として残すため、Spring Batch / Lambda / Lambda + SQS の比較を明記する。
※ 本プロジェクトは **AWS 無料枠完走** を主軸とするため、コスト観点を最優先で評価する。

## ① Spring Batch（Amazia Core 同居案）
- **メリット**
  - Java/Spring エコシステム内で完結、既存の Service/Repository をそのまま再利用可
  - Chunk 指向処理・Step 再開・JobRepository による状態管理が標準装備
  - EC2 上の Core にスケジューラを同居させれば追加コスト 0
- **デメリット**
  - EC2（t2.micro 想定）の CPU/メモリを常時消費。重い処理が走ると Web レスポンスに影響
  - スパイクに弱い（縦スケールしかできない）
- **向く処理**
  - 件数が予測できる定期バッチ（在庫整合性チェック、売上集計、配送ステータス更新）

## ② AWS Lambda（直接トリガー案）
- **メリット**
  - 必要なときだけ起動 → EC2 を軽く保てる
  - 無料枠：100万リクエスト/月 + 400,000 GB秒 のコンピューティング時間
  - 自動スケールするためスパイクに追従
- **デメリット**
  - スパイク時にスケールが追いつかず **スロットリング** が発生する可能性
  - 回避策の Provisioned Concurrency は **常時課金** のため無料枠主義と相性が悪い
  - 最大15分の実行時間制限 → 大量データの一括処理には不向き
  - Java コールドスタートが遅いため、ランタイムは Python / Node.js を検討
- **向く処理**
  - 画像変換、郵便番号データの差分取り込み、通知送信などの単発・短時間処理

## ③ Lambda + SQS（バッファ案）
- **構成**：`EC2 (Amazia Core) → SQS → Lambda`
- **メリット**
  - SQS がスパイクを吸収 → スロットリング回避
  - SQS の可視性タイムアウトと DLQ で **再実行・失敗ハンドリング** が標準化される
  - EC2 は「キューに投げるだけ」で CPU/メモリ消費がほぼ無い
  - SQS 無料枠：100万リクエスト/月（標準キュー）
- **デメリット**
  - 構成要素が増えるため、ローカル開発・テスト環境の整備コストが上がる（LocalStack / ElasticMQ 等）
  - 順序保証が必要な処理は FIFO キューが必要となり、スループット制限あり
- **向く処理**
  - 配送ステータス更新後の通知ファンアウト、トラブル関数のシミュレーション結果通知など、非同期で OK な処理

## 採用方針（暫定）
| 処理種別 | 採用基盤 | 理由 |
|---|---|---|
| 日次の整合性チェック（在庫・売上・配送） | **Spring Batch on EC2** | 件数が予測可能・Core の Service 層を再利用したい・追加コスト 0 |
| 月次の郵便番号データ更新 | **Lambda（EventBridge スケジュール）** | 月1回の単発処理・無料枠で十分・EC2 を巻き込みたくない |
| 通知ファンアウト（メール/Console/Slack） | **Lambda + SQS** | スパイク吸収と再試行を AWS マネージドに任せる |
| トラブル関数（疑似障害） | Spring Batch 内に実装 | ロジックがアプリ仕様と密結合・テスト容易性を優先 |

> 無料枠を超えそうな兆候（CloudWatch のメトリクス）が出たら、Provisioned Concurrency ではなく **バッチ間隔の見直し / 処理対象の絞り込み** で対処する。
