# バッチ処理詳細

---

## 📅 日次バッチ

### ① 商品入荷数と在庫数の整合性チェック
- 入荷テーブルと在庫テーブルを照合  
- 不一致がある場合、対象商品の担当権限者へ通知  
- 通知方法：メール or Console の通知センター（要検討）

---

### ② 発売日・予約ステータスの自動更新
- 発売日・公開日・予約開始日の条件に基づき、商品ステータスを更新  
- フェーズ14・フェーズ15の仕様に基づく  
- 対象ステータス例  
  - 予約開始前  
  - 予約受付中  
  - 公開中  
  - 在庫切れ（予約可）  
- 各フェーズの仕様を再確認し、ロジックを統合する必要あり

---

### ③ 売上と在庫数の照合
- 売上テーブルと在庫テーブルを比較  
- 不一致がある場合、権限者へ通知  
- 売上金額の集計も実施  
- 振込確認の疑似API  
  - ランダム関数で「一致 / 不一致」を返す  
  - 一致 → 正常  
  - 不一致 → NG として通知

---

### ④ 配送ステータスの自動更新
- 配送予定日・発送日・配達完了日を基にステータスを更新  
- トラブル関数（後述）により、一定確率で配送遅延・不在再配達を発生させる

# ⚠ トラブル関数（シミュレーション）

運用テストのため、ランダム関数を用いて「トラブル発生」を再現する。

### ① 売上不一致関数
- 一定確率で売上金額と振込金額を不一致にする  
- 不一致時は権限者へ通知  
- テスト目的で確率は調整可能（例：5%）

### ② 在庫数不一致関数
- 一定確率で在庫数と売上数を不一致にする  
- 不一致時は通知  
- 在庫管理の異常系テストに利用

### ③ 配送トラブル関数
- 宛先不在  
- 配送遅延  
- 再配達  
- 配送ステータスを自動で変更し、Market の購入履歴にも反映
