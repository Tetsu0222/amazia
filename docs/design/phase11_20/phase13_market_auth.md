# 1. 機能概要

- Amazia Market にログイン機能・会員登録機能・パスワード再発行機能を追加  
- HTTPS 化（ALB または CloudFront 経由でアクセス）  
- HTTP → HTTPS リダイレクトを ALB または CloudFront で実施  
- 証明書は ACM を利用し、自動更新を前提とする  
- 郵便番号から住所を自動反映する仕組みを導入  
- Amazia Core に郵便局API連携および住所マスタ登録処理を実装  
- メール送信は Amazon SES を採用し、SPF/DKIM/DMARC を設定  
- AWS 無料利用枠を考慮した構成・代替案を併記

# 2. HTTPS 化（詳細）

## 2.1 要件
- **独自ドメイン利用前提**  
  - Route53 で DNS 管理  
  - ACM 証明書を ALB または CloudFront にアタッチ  
- **HTTP → HTTPS リダイレクト**  
  - ALB リスナー（80 → 443）  
  - または CloudFront の Viewer Protocol Policy（Redirect HTTP to HTTPS）  
- **証明書更新運用**  
  - ACM の自動更新  
  - 更新失敗時は CloudWatch アラート通知

## 2.2 インフラ構成（通常構成）
- Route53 → ALB（443） → ECS(Fargate)  
- WAF（必要に応じて）  
- SES によるメール送信  
- Redis（ElastiCache）でトークン管理  
- CloudWatch Logs / Metrics

## 2.3 CloudFront 非採用理由（明記）
- Market サイトは動的ページ中心であり、キャッシュ効果が限定的  
- ALB 直アクセスの方が構成がシンプル  
- ただし、**無料期間中は CloudFront の方が ALB より安価**  
→ 無料期間限定で CloudFront を利用する代替案を後述

# 3. 認証方式（確定仕様）

## 3.1 方式
- **セッション方式（Cookie ベース）を採用**

## 3.2 Cookie 設定
- `Secure`：true  
- `HttpOnly`：true  
- `SameSite`：Lax  
- 有効期限：30分（操作で延長）

## 3.3 CSRF 対策（追加）
- CSRF トークンをフォームに埋め込み  
- トークンはセッションと紐付けて検証  
- 有効期限：30分  
- Ajax リクエストはヘッダに CSRF トークンを付与

## 3.4 ログイン試行回数制限（ブルートフォース対策）
- 5回失敗で 5 分間アカウントロック  
- CloudWatch + WAF で IP ブロックも検討

# 4. ログイン画面（Amazia Market）

- メールアドレス + パスワードでログイン  
- 認証成功後、ユーザーマイページへ遷移  
- ログイン失敗時はエラーメッセージ表示  
- ログイン成功/失敗はログに記録（個人情報はマスク）

---

# 5. 会員登録画面（Amazia Market）

## 5.1 入力項目
- 名前（姓・名）  
- 住所（郵便番号 → 自動補完）  
- 生年月日（YYYY-MM-DD）  
- メールアドレス  
- 決済方法（プルダウン）  
  - クレジットカード  
  - 銀行振込  
  - その他（DB 管理）

## 5.2 バリデーション
### パスワードポリシー
- 8文字以上  
- 英大文字・英小文字・数字を含む  
- 過去5回分のパスワードは再利用不可

### 生年月日
- 18歳未満は登録不可

### メールアドレス重複チェック
- Ajax で事前チェック  
- DB 側でもユニーク制約  
- エラー文言：「このメールアドレスは既に登録されています」

## 5.3 クレジットカード情報
- **カード番号は保存しない（PCI DSS 準拠）**  
- 外部決済代行サービスの **トークン化方式** を採用  
- DB にはカードトークンのみ保存

## 5.4 郵便番号 → 住所自動反映（UI仕様追加）
- 郵便番号入力後、自動で住所候補を取得  
- 候補が複数ある場合はプルダウンで選択  
- エラー時は「住所が取得できませんでした」と表示

# 6. パスワード再発行フロー

## 6.1 フロー
1. メールアドレス入力  
2. 認証メール送信（ワンタイムリンク）  
3. リンククリックで本人確認  
4. パスワード再設定画面へ遷移  
5. 新パスワードを設定

## 6.2 再認証の定義
- メールリンククリックで本人確認完了  
- ログイン中ユーザーはパスワード入力を要求

## 6.3 トークン仕様
- 有効期限：30分  
- 使い捨て  
- 保存場所：Redis（ElastiCache）  
- 暗号化方式：AES-256  
- トークンはハッシュ化して保存  
- **メール内 URL は独自ドメイン固定（フィッシング対策）**

# 7. Amazia Core 側の実装

## 7.1 郵便局API連携

# 9. AWS 無料利用枠（Free Tier）考慮（新規追加）

## 9.1 無料枠対象外サービスと代替案
| サービス | 無料枠 | 現状 | 代替案 |
|---------|--------|------|--------|
| ALB | ❌ | 必須構成 | CloudFront + Lambda@Edge |
| Fargate | ❌ | 採用 | EC2 t2.micro / t3.micro |
| ElastiCache | ❌ | トークン管理 | DynamoDB TTL / RDS TTL |
| SES | ⭕（EC2経由のみ） | 採用 | EC2 経由で送信 |
| Route53 | △（有料） | 採用 | そのまま |

## 9.2 無料期間中の推奨構成
- CloudFront → S3（静的）＋ Lambda@Edge（動的 API ルーティング）  
- EC2（t2.micro）で API サーバ  
- DynamoDB TTL でトークン管理  
- SES（EC2 経由）でメール送信

# 10. SES 運用（追加）

## 10.1 サンドボックス解除手順
- AWS サポートに申請  
- 利用目的・送信ドメイン・想定送信量を記載  
- 承認後、任意のメールアドレスへ送信可能

## 10.2 送信制限
- レートリミットを CloudWatch で監視  
- バースト時はキューイング（SQS）を検討

# 13. 障害時の復旧手順（追加）

## Redis 障害時
- トークン再発行を許可  
- 一時的に DB バックアップ方式に切替

## SES 障害時
- リトライ（指数バックオフ）  
- 長期障害時は SendGrid にフェイルオーバー

## DB 障害時
- RDS Multi-AZ による自動フェイルオーバー  
- 復旧後は整合性チェック
