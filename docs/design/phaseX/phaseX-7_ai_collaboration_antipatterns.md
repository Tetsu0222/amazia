時系列フェーズ（1〜20）に依存しない横断的品質改善フェーズ。
本プロジェクトは Claude Code を中核に据えた AI 協働開発で進めており、`docs/ai_context/` に

**「AI に任せて失敗したパターン・任せてはいけない判断領域」の知見は未整備** であり、
ポートフォリオとして AI 駆動開発の実践度を語るうえで盲点になっている。

phaseX-6 がスキーマ層の構造的盲点に対する直接対策だったように、本フェーズは
**AI 協働における構造的盲点に対する対策** として位置付ける。

## 背景・なぜ今やるか

### 構造的盲点

```
AI が既存実装を読まずに新規生成
  └─ 人間が差分を流し読みでマージ
      └─ 既存ロジックと衝突 / ドキュメント未追従 / 実機未検証のまま本番投入
          └─ トラブル発生（troubles/NNN として記録）
              └─ 個別の根本原因は記録されるが「AI協働観点での再発防止策」は抽出されない
                  └─ 同型のAI協働ミスが別の場所で再発
```

### 現状

- `docs/ai_context/test_insights.md` — テスト観点の知見（成功事例）
- `docs/ai_context/operational_insights.md` — 実装・運用パターンの落とし穴（成功事例）
- `docs/troubles/` — 50件の不具合記録（個別事例）

troubles の中には AI 起因・AI×人間の協働ミスが一定数含まれるが、
個別 markdown には「AI協働観点」という切り口の節が無いため、
**AI 協働の失敗パターンを横断的に学ぶ手段が無い** 状態になっている。

### なぜ次フェーズで対応するか

AI 駆動開発をポートフォリオの主軸として打ち出すうえで、
「成功知見だけ整備されていて失敗知見が無い」状態は説得力を欠く。
phase16（UI 改善）以降は機能追加が中心になり、新規トラブルが発生する頻度は
phase11〜15 期より下がる見込みのため、**蓄積済みトラブル 50 件を素材として
過去棚卸しを行うのに適したタイミング** でもある。

## 実装計画

### Step 1：軽量タグ付け（Claude 主導・ユーザーレビュー）

**目的：** 50 件の中から AI 協働観点で深掘りすべき候補を絞り込む

**成果物：** `docs/ai_context/_triage_log.md`（一時ファイル）

**タグ定義：**

| タグ | 定義 |
|------|------|
| `[AI起因]` | AI の判断・実装ミスが主因（既存コード未確認・ドキュメント鵜呑み・実機未検証・誤報告など） |
| `[協働ミス]` | 人間の指示曖昧・確認漏れ・スコープ未明示など、人間側の関与で防げた可能性が高い |
| `[AI無関係]` | インフラ起因・外部要因・MUI バージョン差異など AI 協働の改善余地が薄い |

**手順：**

1. Claude が `docs/troubles/001`〜`050` を順に読み、各ファイルにタグ推奨と 1 行の根拠コメントを付けたリストを `_triage_log.md` に出力する
2. ユーザーがレビューし、判定に違和感のあるものだけ修正する
3. `[AI起因]` `[協働ミス]` のうちパターンの異なる代表 5〜10 件を「深掘り対象」としてマークする

**完了条件：**
- `_triage_log.md` に 50 件すべてのタグが記入されている
- 深掘り対象が 5〜10 件選定されている

---

### Step 2：深掘り対象のパターン抽出

**目的：** 個別事例から汎化されたアンチパターンを抽出する

**成果物：** Step 3 のドラフト元になる構造化メモ（`_triage_log.md` 内に追記）

**抽出する観点：**
- 症状（外形的に何が起きたか）
- AI 側の判断ミス（何を読まなかった／何を確認しなかった）
- 人間側の予防機会（どこで止められたか）
- 検知タイミング（CI / レビュー / 本番）
- 同型再発を防ぐためのガード（事前 / 事後）

**完了条件：**
- 深掘り対象の各件について上記 5 観点が埋まっている
- 重複パターンは 1 件に集約済み

---

### Step 3：アンチパターン集の作成

**目的：** AI が将来の作業時に参照できる形でアンチパターンを成文化する

**成果物：** `docs/ai_context/ai_collaboration_antipatterns.md`

**書式案：**

```markdown
# AI協働アンチパターン集

## 目的
本ファイルは Claude Code をはじめとする AI 協働開発において、
過去のトラブルから抽出された「AI 任せで踏みやすい落とし穴」と
「人間側の予防機会」を成文化したもの。
AI は実装・計画着手時に本ファイルを参照し、該当パターンに陥らないこと。

## 本ドキュメントの前提（責任の切り分け方針）

本質的には、AI 協働開発で発生したすべてのミスは人間側の責任である。
AI に何を読ませるか、何を確認させるか、どこで止めるかを設計するのは
人間の役割であり、AI の判断ミスはその設計の不備の現れと捉えるべきである。

しかし「すべて人間が悪い」で片付けると改善のレバーが見えなくなる。
そこで本ドキュメントでは便宜的に以下の3軸で切り分ける：

- **AI 側の判断ミス** — AI が陥りやすい認知の偏り。プロンプト設計や
  コンテキスト供給で対策可能
- **人間側の予防機会** — 指示・レビュー・前提共有のどこで止められたか
- **構造的ガード（事前 / 事後）** — 個人の注意力に依存せず仕組みで防ぐ手段
  （CI / lint / テンプレ / ヘルスチェック等）

この切り分けは責任の所在を決めるためではなく、改善の打ち手を見つける
ための分析フレームワークである。

## アンチパターン一覧

### AP-001: <パターン名>

**症状：**
**AI 側の判断ミス：**
**人間側の予防機会：**
**検知タイミング：**
**事前ガード：**
**事後検知：**
**対応プロンプトスニペット：**
> （AI に渡す 1〜3 行の定型句。本 AP に陥らないための事前チェックを促す内容）
**出典：** docs/troubles/NNN_xxx.md

### AP-002: ...
```

**完了条件：**
- 冒頭に「本ドキュメントの前提（責任の切り分け方針）」節が記載されている
- 5〜10 件のアンチパターンが上記書式で記載されている
- 各 AP に「対応プロンプトスニペット」が併記されている（Step 5 のテンプレ集とも整合）
- すべての項目に出典トラブル番号が紐付いている
- `docs/troubles/README.md` から本ファイルへの導線が張られている

### Step 4：今後の運用ルール組み込み

**目的：** Step 3 の成果物が「死んだドキュメント」にならないよう、新規トラブル発生時に
自動蓄積される運用に組み込む

**変更対象：**
- `CLAUDE.md` の不具合ドキュメントテンプレ
- `docs/troubles/README.md`（運用ルール節）

**CLAUDE.md テンプレへの追加例：**

```markdown
## AI協働観点
- AI の判断ミス：<あれば記載／無ければ「該当なし」>
- 人間が止めるべきだった点：<あれば記載／無ければ「該当なし」>
- 該当アンチパターン：AP-NNN <or 新規パターンとして追加 / なし>
```

**運用ルール：**
- 新規トラブル記録時、`## AI協働観点` 節の埋め込みを必須化
- 既存の AP-* に該当しない新規パターンが現れた場合は、トラブル対応完了時に
  `ai_collaboration_antipatterns.md` に AP-NNN として追記する

**完了条件：**
- CLAUDE.md のテンプレ更新が反映されている
- `docs/troubles/README.md` の運用ルール節に AI 協働観点の記録ルールが追記されている

### Step 5：プロンプトテンプレート集の整備

**目的：** アンチパターン集を「読み物」で終わらせず、実装着手時に AI へ渡せる
定型プロンプトとして実用化する。AI 駆動開発のポートフォリオ価値を一段引き上げる。

**成果物：** `docs/ai_context/prompt_templates.md`

**設計方針：**
- Step 3 の AP-* と一対一対応にせず、**作業種別**（新規テーブル追加 / ライブラリ採用 /
  バグ修正 / フェーズ完了確認 等）で章立てする
- 1 つの作業種別に複数 AP が関連することが多いため、各テンプレ末尾に「関連 AP-NNN」を併記
- AI が Copy-Paste でそのまま使える形式にする（変数部分は `<...>` で明示）

**書式案：**

```markdown
# プロンプトテンプレート集

## 目的
作業種別ごとに、AI へ渡す定型プロンプトを集約する。
本ファイルのテンプレを使うことで、`ai_collaboration_antipatterns.md` の
AP-* に該当する典型的な落とし穴を着手前に予防できる。

## テンプレ一覧

### TPL-001: 新規テーブル追加時
> <テーブル名> を追加する前に：
> 1. schema.sql を読み、参照先となる既存テーブル（特に users.id）の型を確認
> 2. FK 列が同一型（BIGINT UNSIGNED）になっているか
> 3. 既存の同概念テーブルがないか grep
> を順に実行し、結果を報告してから DDL を書いてください。
>
> 関連 AP: AP-001

### TPL-002: ライブラリ採用時
> <ライブラリ名> の <機能> を使う前に、公式ドキュメントの該当節を引用し、
> 以下を報告してください：
> - 採用するメソッド/クラスの正確なシグネチャ
> - デフォルト挙動（特にデフォルト無効になっている設定）
> - 制約（鍵長・型・バージョン依存など）
>
> 関連 AP: AP-004

### TPL-003: ...
```

**最低限揃えるテンプレ（叩き案）：**

| TPL | 作業種別 | 関連 AP |
|-----|----------|---------|
| TPL-001 | 新規テーブル追加時 | AP-001 |
| TPL-002 | ライブラリ採用時 | AP-004 |
| TPL-003 | テストヘルパー作成時 | AP-002 |
| TPL-004 | マイグレーション機構の確認 | AP-003 |
| TPL-005 | シェル/SQL 文字列構築時 | AP-005 |
| TPL-006 | バグ修正時の横展開確認 | AP-006 |
| TPL-007 | フェーズ完了確認時 | AP-007 |
| TPL-008 | CI/CD・運用機構導入時 | AP-008 |

**完了条件：**
- `prompt_templates.md` が新設され、最低 8 件のテンプレが収録されている
- 各テンプレに「関連 AP-NNN」が併記されている
- `ai_collaboration_antipatterns.md` の各 AP からも `prompt_templates.md` の該当 TPL への
  双方向リンクが張られている
- README または CLAUDE.md からテンプレ集への導線が張られている

## リスクと対策

| リスク | 対策 |
|--------|------|
| Step 1 のタグ付けが主観的になる | タグ定義を明文化し、根拠コメントを 1 行併記する／ユーザーレビューで揺らぎを補正 |
| 抽出したアンチパターンが汎化しすぎて実用性を欠く | 出典トラブル番号を必ず併記し、具体例から離れすぎないようにする |
| Step 4 の運用ルールが形骸化 | 新規トラブル記録時の必須節として CLAUDE.md に組み込み、AI が記入忘れを検知できる構造にする |
| 差し込み実行で中断 → Step 2 以降が放置 | 各 Step が独立完結する設計のため、Step 1 の `_triage_log.md` までで停止しても再開可能 |
| Step 5 のテンプレ集が AP と乖離して保守されなくなる | AP と TPL を双方向リンクし、AP 追加時に対応 TPL の有無を確認する運用ルールを CLAUDE.md に明記 |
| 「責任はすべて人間」の前提が責任転嫁批判を呼ぶ | 冒頭で「分析フレームワークであって責任の所在を決めるものではない」と明示する |
