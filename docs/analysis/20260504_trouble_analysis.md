## 詳細分析

### 007 — Excel一括インポート 422エラー（テスト不足 + 環境設定）

**分類：テストケース不足 ＋ 環境設定上の課題**

原因が3層に分かれる複合障害。

1. **テスト不足（原因1）**：`a-upload-dragger` の `.originFileObj` 構造はフロントエンドの結合テストがあれば検出できた。Vue コンポーネントレベルのテストが存在しなかった。
2. **環境設定（原因2）**：`mimes` バリデーションが Alpine Linux の `libmagic` 不完全性により壊れる。006と同じ「ローカル環境と Docker 環境の乖離」パターンの再現。
3. **テスト不足（原因3）**：ルート順序の問題は PHPUnit でルーティングテストを書いていれば検出可能だった。

**再発防止**：Ant Design アップロード系コンポーネントの `.originFileObj` 取り出しを coding_guidelines に標準化する。Laravel の `mimes` バリデーションはコンテナ環境で不安定なため `mimetypes` による明示指定をデフォルトにする。

### 008 — EC2再起動後コンテナ自動起動しない（環境設定）

**分類：環境設定上の課題**

`docker-compose.yml` に `restart: unless-stopped` が設定されておらず、EC2 再起動後にコンテナが Exited のまま残った。001・003・004と同じ「AWSインフラ固有の挙動の未把握」パターン。

- テストで検出できる性質ではない
- デプロイ後ヘルスチェック（003・005の再発防止アクション）が実施済みであれば即時検出できた

**注目点**：003・005の再発防止アクションが本不具合発生時点で未実施だったことを間接的に示している。

**再発防止**：`docker-compose.yml` へのサービス追加時に `restart: unless-stopped` を必須とするチェックリスト項目を追加する。

---

### 009 — CORE_BASE_URL未設定による500エラー（テスト不足）

**分類：テストケース不足（最も典型的なケース）**

CI はグリーンなのに本番 500 という「テストが意味をなしていない」状態。原因は明確に2つ。

1. **テストの URL ハードコード**：`Http::fake(['http://localhost:8080/...'])` が実際の環境変数と無関係に通過する。テストと実行環境の URL が乖離していた。
2. **環境変数管理の不整合**：`CORE_BASE_URL` を新規追加した際に `docker-compose.yml` と `phpunit.xml` のセット更新が行われなかった。

`feedback_env_vars_and_tests.md` に記録された教訓（新規環境変数は docker-compose.yml と phpunit.xml をセットで更新・テストは config() 経由で）はこの不具合から生まれている。

**再発防止**：テストの URL を `config()` 経由に統一し、phpunit.xml の値を検証対象にする。新規環境変数追加を PR テンプレートのチェックリストに明記する。

### 010 — Market商品未表示（環境設定）

**分類：環境設定上の課題**

`docker-compose.yml` に `amazia-market` サービス定義が存在しなかった。008と完全に同根（インフラ定義漏れ）。

- フロントエンド（Vite）のため単体テストがなく CI では検出不可
- 「新サービスを実装したら docker-compose.yml への追加が必要」という手順がチェックリスト化されていなかった
- 003・005・008で繰り返されてきた「デプロイ後検証不足」パターンの継続

**再発防止**：新サービス追加時の PR チェックリストに「docker-compose.yml への追加」と「デプロイ後ヘルスチェック対象への追加」を含める。

---

### 011 — Consoleサイドバー未表示（テスト不足）

**分類：テストケース不足（フロント実装漏れ）**

フェーズ10の設計ドキュメントには明記されていたが、`App.vue` のサイドバーレイアウト実装と `router/index.js` へのルート登録が行われていなかった。

- PHPUnit（バックエンド）では検出できない Vue の実装漏れ
- E2E テストまたはデプロイ後のスクリーンショット確認があれば即時検出できた
- 003・005で指摘された「デプロイパイプラインの検証不足」が Vue 側でも同様に存在している

**再発防止**：新機能追加時のチェックリストに「App.vue・ルーター更新」を追加する。デプロイ後にスクリーンショット確認ステップを CI に組み込む。

---

### 012 — 公開期間がMarketに反映されない（テスト不足 + 環境設定）

**分類：テストケース不足 ＋ フェーズ移行時の設計乖離**

最も構造的な問題を含む不具合。二重の問題が存在する。

1. **テスト不足（問題A）**：`ProductForm.vue` の price/stock 必須バリデーションは「公開期間のみ変更する」ユースケースのテストがあれば検出できた。
2. **テスト不足（問題B）**：`Product.java` の `@NotNull` 残留は、price/stock を送らないリクエストの統合テストを書いていれば検出できた。
3. **根本は設計乖離**：フェーズ10の「price/stock を SKU に移行する」という設計変更が、エンティティ・フォーム・バリデーションに一貫して適用されていなかった。

**フェーズ移行時に廃止フィールドを同時に削除するという規律がなかったこと**が根本原因。

**再発防止**：フェーズ移行時に廃止フィールドの一覧を PR 説明に記載し、レビュアーが確認する。「公開期間を設定 → Market に表示されるか」を手動テストチェックリストに追加する。

---

### 013 — Console画像登録の導線なし（テスト不足）

**分類：テストケース不足（バックエンド先行実装のデプロイ）**

API は TDD グリーン済みだが Console フロントエンドに画像アップロード UI が追加されていなかった。

- バックエンドの TDD グリーン ≠ 機能完成 であることを示す典型的な事例
- フロントエンドの結合テスト・E2E テスト、あるいは「UI 導線が存在するか」のデプロイ前チェックがあれば防げた
- 011とほぼ同根（フロント実装漏れ + 検証不足）

**再発防止**：バックエンド API 実装 PR のレビューチェックリストに「UI 導線が存在するか」を追加する。Console の各機能ページをブラウザで操作し、スクリーンショットを PR に添付することを標準化する。
