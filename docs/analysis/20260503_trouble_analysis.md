## 詳細分析

### 001 — SSM ConnectionLost（環境設定）

**分類：環境設定上の課題**

`reboot` と `stop/start` の挙動の違いという AWS インフラ固有の知識不足が原因。SSM エージェントが異常終了した状態で `reboot` しても状態が持ち越される仕様を知らなかったことによるもの。

- テストで検出できる性質の問題ではない
- 事前にAWS公式ドキュメントやベストプラクティスを参照することで防げた

**再発防止**：運用手順書に「インスタンス復旧は stop/start を基本とする」と明記する。

#### 追記（2026-05-03）

上記の分析は「SSMエージェントが異常終了した**後の復旧手順**」を記録しているに過ぎず、**なぜ異常終了したのか**という根本原因の追究が抜けている点に課題がある。

現状の記録構造は「症状 → 回避策」であり、本来は「症状 → 原因 → 根本原因 → 対策」の構造であるべき。

追究すべきだった観点：

- `reboot` のトリガーは何だったか（意図的な操作か、OSパニック・OOM等の異常か）
- `reboot` 前後で SSM エージェントのログ（`/var/log/amazon/ssm/`）を確認したか
- SSM エージェント自体のバージョンや systemd の Restart 設定に問題がなかったか

根本原因が不明なままでは、stop/start による復旧はあくまで回避策であり、同じ異常終了が再発した場合に同様のトラブルが繰り返される可能性がある。次回同様の事象が発生した際はログ調査を必ず実施する。

#### 追記（2026-05-03）— 根本原因の特定・解決済み

`/var/log/amazon/ssm/amazon-ssm-agent.log` の調査により、`docker-compose up --build` の長時間実行によるIPCタイムアウトとメモリ逼迫（`signal: killed`）の累積が根本原因と判明。「意図しないreboot」ではなくCI/CD設計ミスによる障害だった。003の対応（ビルドをCI側に移行）が恒久対策となる。001_ssm_connection_lost.md を更新済み。

### 002 — MySQL host not allowed（環境設定）

**分類：環境設定上の課題**

`MYSQL_USER` で作成されるユーザーのホスト制限（`localhost` vs `%`）というDocker環境固有の初期化挙動が原因。既存ボリュームが残っていると環境変数の再適用が行われないという Docker の仕様も絡んでいる。

- コンテナ間通信の接続テストをローカルで実施する手順があれば早期に検出できた
- ただし根本は環境設定の問題（ボリューム管理ポリシーの欠如）

**再発防止**：開発環境セットアップ手順に「初期化時は必ず `-v` でボリュームを削除する」を明記する。

---

### 003 — SSM コマンドキュー詰まり（環境設定 + テスト不足）

**分類：環境設定上の課題 ＋ テストケース不足**

2つの原因が複合している。

1. **環境設定**：`send-command` が非同期であることを前提にしない設計（完了待機なし）
2. **テスト不足**：CI/CDパイプラインの結合テストが不足していた。「Actions上では成功・EC2では未実行」という状態がテストによって検出されていなかった

- デプロイ後のスモークテスト（エンドポイントへのヘルスチェック等）があれば失敗を即時検出できた
- SSM の非同期挙動を考慮したパイプライン設計レビューが必要だった

**再発防止**：デプロイ後にアプリのヘルスチェック用ステップを Actions に追加する。SSM コマンドはポーリング待機を標準化する。

---

### 004 — EC2 IP変更によるCORSエラー（環境設定）

**分類：環境設定上の課題**

AWSのパブリックIPが動的である仕様の未考慮と、CORS の `allowedOrigins` にIPをハードコードしていた設計ミスが原因。

- ユニットテストでは検出不可能（本番のIP変動はテスト環境では再現できない）
- 環境変数または設定ファイルでCORSオリジンを管理する設計にすれば、コード変更なしに対応できた

**再発防止**：`allowedOrigins` を環境変数化し、ハードコードを排除する。Elastic IP の割り当てを恒久対応として実施する。

---

### 005 — Nginx 403 Forbidden（環境設定 + テスト不足）

**分類：環境設定上の課題 ＋ テストケース不足**

2つの原因が複合している。

1. **環境設定（003の波及）**：SSMキュー詰まりによりdistコピーが未実行だった（003参照）
2. **テスト不足**：`vite.config.js` の `outDir` 設定と deploy.yml のコピー元パスが乖離していたことが、デプロイ前の検証で検出されていなかった

- デプロイスクリプト内で `ls` によるファイル存在確認ステップを入れていれば即時検出できた
- ViteのビルドパスとデプロイスクリプトのパスをIaCまたはCI設定で一元管理することで防げた

**再発防止**：zipコマンド実行前に `ls public/vue/` で成果物の存在を確認するステップを追加する。

---

### 006 — Composer platform ext missing（環境設定）

**分類：環境設定上の課題**

ローカル（XAMPP）の PHP 拡張設定と、Docker・CI の実行環境の乖離が原因。Composer がデフォルトでローカル環境を本番相当と見なす挙動への理解不足も絡む。

- ローカルでの `composer update` 結果がそのままCIに反映されないことが問題の本質
- CI上で事前に依存インストールのドライランを行うか、`--ignore-platform-reqs` の使用ポリシーを標準化することで防げた

**再発防止**：新しいComposerパッケージ追加時は、CI上での動作確認を必須ステップとして手順書に明記する。

## 全体傾向と考察

### テストケース不足に起因するもの（2件）

| # | 内容 |
|---|------|
| 003 | デプロイ後のスモークテスト・ヘルスチェックの欠如 |
| 005 | デプロイスクリプト内のファイル存在確認の欠如 |

いずれも「アプリケーションロジックのテスト」ではなく、**デプロイパイプラインの検証テスト**が不足していたケース。ユニットテストや機能テストでは検出が困難で、インテグレーションテスト／スモークテストの整備が有効。

### 環境設定上の課題（全6件共通）

全トラブルに共通する環境設定起因の問題は以下の3パターンに集約される。

| パターン | 該当 |
|---------|------|
| AWSインフラ固有挙動の未把握（動的IP・SSM非同期・reboot仕様） | 001, 003, 004 |
| ローカル環境と実行環境（Docker/CI）の乖離 | 002, 006 |
| デプロイスクリプトのパス・設定のハードコード・不整合 | 004, 005 |

## 推奨アクション

1. **デプロイ後スモークテストの追加**（003・005対策）
   - GitHub Actions に「デプロイ完了後、HTTP 200 を確認するヘルスチェックステップ」を追加する

2. **CORS オリジンの環境変数化**（004対策）
   - `allowedOrigins` をハードコードせず、Spring の `application.properties` または環境変数から読み込む

3. **Elastic IP の恒久割り当て**（001・004対策）
   - stop/start による IP 変動を根本的に排除する

4. **デプロイスクリプトへのファイル存在確認ステップの追加**（005対策）
   - `zip` 前・`cp` 前に `ls` で成果物の存在を確認するステップを標準化する

5. **開発環境セットアップ手順書の整備**（002・006対策）
   - Docker ボリューム管理ポリシーと Composer 拡張管理の注意事項をドキュメント化する
