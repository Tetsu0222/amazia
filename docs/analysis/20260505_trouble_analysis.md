## 詳細分析

### 014 — SKU価格管理ページの未実装プレースホルダー残留（テスト不足）

**分類：テストケース不足（フェーズ完了基準の欠落）**

バックエンド（Laravel + Spring Boot + DB）はフェーズ10で完全実装済みなのに、`SkuPriceList.vue` がプレースホルダーのまま残っていた。011・013と完全に同根（バックエンドTDDグリーン ≠ 機能完成）。

- フェーズ10の完了基準が「バックエンドAPIテストグリーン」のみで、UI動作確認が含まれていなかった
- プレースホルダーが残ってもビルド・ルーティングは通るため CI で検知不可
- ブラウザでメニュー項目をクリックする最小限のスモークテストがあれば即時検出できた

**再発防止**：フェーズ完了基準に「新規メニュー項目を実機ブラウザでクリック→画面が機能することをスクリーンショット添付」を必須化する。プレースホルダー利用時は `TODO:` コメントとPRチェックリストでの消し込みをセットにする。

### 015 — amazia-market の ECR pull 失敗（環境設定）

**分類：環境設定上の課題**

amazia-market は Vite 静的配信で Docker 管理外という方針なのに、Nginx 設定マウントの目的で `docker-compose.yml` に再追加され、存在しない ECR リポジトリを pull して失敗した。

- 010（Market商品未表示・docker-compose定義漏れ）と表裏一体の問題：今回は逆に「定義してはいけないものを定義した」
- amazia-market の配信構成が docker-compose.yml のコメントや README に明文化されていなかった
- ECR リポジトリの存在を確認せずに編集・コミットした

**再発防止**：`docker-compose.yml` の冒頭にサービス分担方針（amazia-market は Nginx 静的配信、コンテナ管理外）をコメントで明記する。ECR 参照を追加する際は事前に `aws ecr describe-repositories` で存在確認するチェックリストを設ける。

### 016 — EC2上で SKU画像エンドポイントが404（環境設定）

**分類：環境設定上の課題（デプロイ運用の不備）**

EC2 上の Docker イメージが古く、SKU画像機能のコードが含まれていなかった。amazia-console と amazia-core を片方ずつデプロイしてしまう構造的問題。

- 手動デプロイのため amazia-console / amazia-core の同時更新が保証されない
- 副次的に EC2 ルートディスクが満杯（残り 28MB）になっており、pull そのものが途中失敗
- 旧イメージ48個・ビルドキャッシュ1.165GB が放置されていた（→ 017の伏線）

**再発防止**：amazia-console と amazia-core を1つの GitHub Actions ワークフローでまとめてビルド＆プッシュ＆EC2 pull するよう統合する。EC2 上で `docker system prune` を週次 cron 化する。デプロイ後は両レイヤー（8000/8080）に直接 curl して 404 がないことを確認する。

### 017 — 不要AWSリソースによる想定外課金（環境設定）

**分類：環境設定上の課題（コスト監視の不在）**

無料期間中に $133 のクレジットが消費されていた。016で発見された ECR イメージ蓄積（無料枠500MB超過）がここでもコストとして顕在化。

- ECR ライフサイクルポリシーが未設定で、タグなしイメージが各リポジトリ29個蓄積
- t3.small（無料枠対象外）を本番機として運用していた
- 無関係インスタンス（testSample2）が無料枠750時間/月を二分割消費
- 停止中インスタンスの EBS が 30GB 無料枠を超過
- AWS Billing アラート・Cost Explorer 確認の運用がなかった

**再発防止**：ECR ライフサイクルポリシー（タグなしを1日後削除）を全リポジトリで標準化する。Billing アラート（月 $1 / $5 / $10）を設定する。学習用途は t3.micro 基本、t3.small 一時利用は phaseX-4 完了後に判定する方針を運用に取り込む。

> 関連メモリ：[project_phaseX4_t3micro_recovery.md] — t3.small は phaseX-4 完了まで継続。

### 018 — permissionsテーブル不在によるcore起動失敗（テスト不足 + 環境設定）

**分類：テストケース不足 ＋ 環境設定上の課題**

3層に絡んだ複合障害。007と同様の複合パターン。

1. **テスト不足（問題A）**：H2 インメモリでは `permissions` テーブルが H2 用 data.sql で作られるため CI は通るが、Docker MySQL での初回起動はテストされていなかった
2. **環境設定（問題B）**：`spring.jpa.defer-datasource-initialization=true` が `application-local.properties` のみで `application.properties`（共通）に未設定
3. **設計漏れ（問題C）**：`Permission`/`RolePermission` Entity が存在せず、`ddl-auto=update` でも自動生成されなかった
4. **DB混在（問題D）**：Laravel migration 由来の `users.id`（bigint unsigned）と JPA 自動生成の FK（bigint signed）の型不一致

001〜017の中で初めて「Laravel と Spring JPA が同じ DB を共有する」構造的問題が顕在化した事例。

**再発防止**：フェーズ完了時に `docker compose down -v && docker compose up --build` で DB初期化からの起動を必ず確認する。テーブル定義 SQL に対応する JPA Entity をセットで作成するルールを coding_guidelines に追加する。Spring の共通設定は必ず `application.properties` 側に置く。

### 019 — Consoleログイン500 / Market商品取得401（テスト不足）

**分類：テストケース不足（API契約・公開ルート設計の検証不足）**

二重のテスト空白。

1. **問題A（API契約の未検証）**：Guzzle `SetCookie` の `isSecure()/isHttpOnly()` という存在しないメソッドを呼んでおり、ログイン時に必ず 500。Guzzle と直接接続する統合テストがあれば即時検出できた
2. **問題B（公開ルート設計）**：`/products/market` が `auth.jwt` グループ内に include されていた。マーケット向け公開エンドポイントを未認証で叩く E2E テストがあれば即時検出できた
3. **副次（コンテナ反映）**：イメージ COPY 方式のため、ホスト編集後に `docker compose build` または `docker cp` が必要

**再発防止**：Guzzle の Cookie API は `getSecure()/getHttpOnly()` を使う規約を coding_guidelines に明記する。公開エンドポイントは `routes/api.php` の auth グループ外に明示的に配置する規約を設け、PHPUnit で「未認証で 200 が返るか」のテストを追加する。

### 020 — refresh_token Cookieが保存されない（テスト不足）

**分類：テストケース不足（Cookieライフサイクルの未検証）**

amazia-core が発行する `Set-Cookie: domain=amazia-core`（コンテナ名）を amazia-console がそのままブラウザに転送していたため、ブラウザでは `localhost` と一致せず Cookie が保存されない。

- 015・019・020と続く「Console が Core の出力をどう変換すべきか」というプロキシ層の設計が未整理
- API テストは Bearer トークン直接付与で検証しており、Cookie 経由のリフレッシュフローが一度もテストされていなかった
- 「ログイン → 15分待つ → 自動リフレッシュされる」というシナリオが E2E でカバーされていない

**再発防止**：Core から受け取った Cookie を Console が転送する際は `domain=null` を必須とするヘルパー関数を作る。Core 側で Cookie の `domain` を空にして発行する（リバースプロキシ層で上書きする原則）。Playwright 等で「ログイン→リロード→アクセストークン期限切れ→自動リフレッシュ」のフローを E2E テスト化する。
